# Research-spike snapshot script (`hp-6v3`)

One Bash + jq script that captures the full ACFS toolchain state on a research-spike VPS as a single normalized JSON document. Drives parser fixtures (`plan.md` §16, §18.3, [`hp-pl5o`](#)), the Mock Flywheel corpus (`hp-wle`), the capability registry (`plan.md` §2.8, [`hp-r33`](#)), and per-tool integration contracts ([`hp-78m`](#)).

## Layout

```
scripts/research-spike/
├── README.md                      ← you are here
├── snapshot.sh                    ← entry point (chmod +x)
├── lib/
│   └── envelope.sh                ← ToolCapture + InvocationCapture envelope helpers
└── schema/
    └── snapshot.schema.json       ← JSON Schema (draft 2020-12) for the output
```

The output document validates against `schema/snapshot.schema.json`. Every field documented in the schema is the contract — adapter contract tests (§18.3) and the capability registry (§2.8) consume the same shape.

## Quick start

### Run on a fresh ACFS VPS

```bash
ssh <vps> bash -s -- --scenario fresh \
  < scripts/research-spike/snapshot.sh \
  > packages/fixtures/phase0-2026-05-02/scenarios/fresh/snapshot.json
```

> **Note.** `bash -s --` reads the script over stdin so the VPS doesn't need a copy on disk. The required `lib/envelope.sh` is currently sourced by relative path; for the over-stdin pattern, copy the whole `scripts/research-spike/` tree first via `rsync`/`scp` or invoke the canonical `bash scripts/research-spike/snapshot.sh` after sync.

### Run in-place

```bash
scripts/research-spike/snapshot.sh \
  --scenario active \
  --vps-id research-spike-2026-05-02 \
  --acfs-tag v0.42.1 \
  --fixtures-version phase0-2026-05-02 \
  --output snapshot.active.json
```

### Local self-test (no VPS needed)

```bash
scripts/research-spike/snapshot.sh --self-test --scenario ad_hoc \
  --output /tmp/snapshot.local.json
```

`--self-test` tolerates missing tools — every capture path runs, but absent binaries get `{present: false, skipReason: "..."}` instead of erroring out.

## Scenarios (fixture taxonomy)

The bead requires three captured states (per `plan.md` §16 task 2 and `hp-pl5o`):

| Scenario  | When to use                                                                                                                      | Fixture path                                                  |
| --------- | -------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| `fresh`   | Brand-new VPS right after `acfs onboard` finishes. No swarm running, no in-flight beads, no pending mail.                        | `packages/fixtures/phase0-<date>/scenarios/fresh/`            |
| `active`  | Mid-swarm: at least 2 NTM panes alive, beads in `in_progress`, pending Agent Mail messages, partial git WIP, build queue active. | `packages/fixtures/phase0-<date>/scenarios/active/`           |
| `failure` | Deliberately broken state: `br` DB locked, `bv` cycle present, wedged NTM pane, stale reservation, oversized log, etc.           | `packages/fixtures/phase0-<date>/scenarios/failure/`          |
| `mock`    | Generated by Mock Flywheel Mode (synthetic). Used for unit/integration tests that don't need real-VPS ground truth.              | `packages/fixtures/scenarios/<scenario-id>/`                  |
| `ad_hoc`  | One-off captures for debugging.                                                                                                  | not committed                                                 |

Scenario taxonomy and per-scenario expected-outcomes are spec'd in [`hp-wle`](../../docs/research-spike/research-spike-notes.md#scenario-checklist).

## CLI flags

| Flag                          | Meaning                                                                                            |
| ----------------------------- | -------------------------------------------------------------------------------------------------- |
| `--scenario {fresh\|active\|failure\|mock\|ad_hoc}` | Required for fixture committal; defaults to `ad_hoc`.                                              |
| `--scenario-notes <text>`     | Free-form notes captured into `meta.scenarioNotes`. Use to record what was deliberately broken.    |
| `--vps-id <id>`               | Stable VPS identifier for `meta.vpsId`. Defaults to `hostname`.                                    |
| `--output <path>`             | Write to file instead of stdout. The byte count is logged to stderr.                               |
| `--max-bytes <n>`             | Per-stream cap for stdout/stderr (default 1 MiB). Larger streams are truncated and `truncated:true` is set. |
| `--no-redact`                 | Disable `ENVELOPE_REDACT` (bearer tokens, SSH passphrases, provider creds). Use only for offline parser dev. |
| `--only <tool[,tool...]>`     | Restrict captures to listed tools. See `ALL_TOOLS` in `snapshot.sh` for the full set.              |
| `--skip <tool[,tool...]>`     | Inverse of `--only`.                                                                               |
| `--self-test`                 | Tolerate missing tools; useful on dev workstations.                                                |
| `--schema-validate`           | Require `ajv-cli` (`npm i -g ajv-cli ajv-formats`); validate the output document before emitting.  |
| `--acfs-tag <tag>`            | Records `meta.acfsTag` (`plan.md` §6.3 production-pinning).                                        |
| `--fixtures-version <tag>`    | Records `meta.fixturesVersion` (`plan.md` §2.8). Use the same tag as the directory name.           |
| `-h`, `--help`                | Print this help.                                                                                    |

### Environment variables

| Env                          | Default     | Purpose                                                                  |
| ---------------------------- | ----------- | ------------------------------------------------------------------------ |
| `ENVELOPE_MAX_BYTES`         | `1048576`   | Same as `--max-bytes`.                                                   |
| `ENVELOPE_TIMEOUT_S`         | `30`        | Per-invocation wall-clock cap.                                           |
| `ENVELOPE_REDACT`            | `1`         | `0` disables redaction (same as `--no-redact`).                          |
| `HOOPOE_SNAPSHOT_STRICT`     | `0`         | `1` exits non-zero if any tool capture function fails (off by default).  |
| `HOOPOE_SNAPSHOT_COMPACT`    | `0`         | `1` emits compact JSON; `0` pretty-prints.                               |

## Captured tools

`ALL_TOOLS=(git br bv ntm agent_mail ru health caut caam dcg casr ubs jsm jfp oracle pt srp sbh)`

For each tool, `snapshot.sh` calls a `capture::<slug>()` function that:

1. Probes presence + version (`envelope::probe_version`).
2. Declares capability IDs (`envelope::add_capability`) — mirrors `plan.md` §2.8.
3. Runs documented invocations (`envelope::run_capture`).
4. Closes the envelope (`envelope::tool_close`), which flushes the per-tool JSON to FD 3.

The bead's "REQUIRED COVERAGE" checklist maps to specific captures inside each `capture::<slug>` body — open `snapshot.sh` and grep for the tool name to see what it does.

### Hard guardrails baked into the captures

These are non-negotiable per `plan.md` Appendix C and `AGENTS.md`:

- **No bare `bv`.** Only `--robot-*` surfaces are invoked. Bare `bv` would launch a TUI and block; `bv.tui` capability is reported as `blocked-by-policy` to make the rule visible in the registry. (Guardrail 1)
- **No mutating commands.** `caam switch-account`, `pt kill`, `sbh cleanup`, and `dcg execute` are *never* invoked. The script captures their `--help` shape only and marks the destructive capability as `blocked-by-policy`.
- **No provider SDKs.** No `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` / `GEMINI_API_KEY` env reads, no API calls. (Guardrail 11)
- **Repo never written.** No `git commit` / `git push` / file writes inside the cloned repo. The snapshot script is read-only.
- **Redaction by default.** Bearer tokens, SSH passphrases, provider keys, and PEM private keys are scrubbed via `__ENVELOPE_REDACT_PATTERNS` before bytes are committed to the envelope. `--no-redact` exists for offline parser-development only.

## Running on the research-spike VPS

The first canonical run is part of `hp-r7i` (provision the VPS) — that bead is human-only because it requires a Contabo/OVH account or an existing VPS. After hp-r7i is done:

```bash
# 1. Sync the script over.
rsync -azP scripts/research-spike/ <vps>:/tmp/hoopoe-snapshot/

# 2. Capture all three scenarios.
ssh <vps> 'cd /data/projects/<demo-project> && \
  bash /tmp/hoopoe-snapshot/snapshot.sh \
    --scenario fresh \
    --acfs-tag <pinned-tag> \
    --fixtures-version phase0-2026-05-02 \
    --vps-id research-spike-2026-05-02 \
    --output /tmp/snapshot.fresh.json'
ssh <vps> 'cd /data/projects/<demo-project> && \
  bash /tmp/hoopoe-snapshot/snapshot.sh --scenario active \
    --acfs-tag <pinned-tag> \
    --fixtures-version phase0-2026-05-02 \
    --output /tmp/snapshot.active.json'
ssh <vps> 'cd /data/projects/<demo-project> && \
  bash /tmp/hoopoe-snapshot/snapshot.sh --scenario failure \
    --scenario-notes "broke br DB by chmod 000; wedged ntm pane via SIGSTOP" \
    --output /tmp/snapshot.failure.json'

# 3. Pull and commit.
scp <vps>:/tmp/snapshot.{fresh,active,failure}.json \
    packages/fixtures/phase0-2026-05-02/scenarios/<scenario>/snapshot.json
```

## Known limitations / future work

- **`agent_mail --version`** is not standard; `probe_version`'s fallback returns the help-banner border. Captured for `hp-d54` gotchas.
- **`lizard --xml`** can be slow on monorepos; `TIMEOUT_S=60` may need to be raised. `health` capture is best-effort.
- **`ubs --ci .`** scans the whole tree; may need to be scoped per scenario.
- **NDJSON detection** in `envelope::run_capture` is conservative — it only treats stdout as JSON when *every* line parses. Mixed-mode streams fall back to `stdoutText` only.
- **Idempotency** (Phase 0 acceptance: byte-identical output on repeat runs) is partial — timestamps, durations, and host stats vary by definition. Adapter contract tests should compare on a normalized projection (`jq 'del(..|.durationMs?, ..|.capturedAt?, ..|.host?)'`). The reproducibility test is part of `hp-pl5o`.

See `docs/research-spike/research-spike-notes.md` for the field notebook captured during real-VPS runs and `docs/research-spike/gotchas.md` (`hp-d54`) for the structured gotcha catalog.
