## TL;DR

Hoopoe is on track as a Phase 1 scaffold, but it does **not** yet pass the plan's Phase 0 or Phase 1 acceptance tests end-to-end.

Before Phase 2 feature build-out, run a short Phase 1.5 acceptance-closure pass focused on real evidence, not more scaffold.

**Verdict: Done.**

## 6. Mock Flywheel Mode (§13)

**Verdict: Partial.**

Mock Flywheel has real backend-side pieces: fixture replay loader, mock daemon client, sequence cursors, mock auth tokens, Local Demo bootstrap, scenario swap, and unit tests. `apps/desktop/src/main/MockFlywheelMode.test.ts`, `LocalDemoBootstrap.test.ts`, and hp-j30 unit tests prove those pieces work without a VPS.

It is not yet full end-to-end product proof. Renderer stages still mostly render static placeholders, and Playwright either fails on host dependencies or skips. A user can open the desktop shell against fixture-labeled projects, but they cannot yet exercise the four stages as data-backed cockpit surfaces from the fixture corpus.

## 7. CI/CD and Packaging

**Verdict: Partial.**

The release pipeline exists and is mac-only (`macos-14`, arm64/x64). It installs, runs preflight checks, builds DMG/ZIP artifacts, uploads assets, and publishes through GitHub Releases. The build script contains signing/notarization config behind `--signed`; release workflow now fails if signing secrets are absent.

What is missing:

- No evidence that Apple signing/notarization has actually run.
- No root `dist:desktop:dmg:arm64` script matching plan wording.
- No separate PR CI workflows for typecheck/lint/test beyond the release workflow and local scripts.
- `bun run e2e` is missing at repo root.
- Mock update server is unit-smoked, not release-smoked through an actual built app update flow.
