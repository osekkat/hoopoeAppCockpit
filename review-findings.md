## Phase 1.5 cross-review — [HIGH] Multi-key settings rollback can leave false audit entries
**Where:** `apps/desktop/src/main/SettingsBridge.ts:510`   **Issue:** `auditDelta()` writes audit rows one at a time before the settings file is written. If a multi-key settings update changes two security-relevant keys and the sink accepts the first row but rejects the second, `setUserSettings()` throws and rolls back the settings write, but the first audit row already claims a setting changed when it did not. That breaks the audit log's source-of-truth contract.   **Suggested fix:** Precompute the full redacted entry batch, then deliver it through a synchronous transactional/batch sink that either accepts all entries before disk mutation or writes none. Add a regression test with two security keys and a sink that fails on the second entry; assert no audit rows are committed and settings remain unchanged.   **Reviewer:** p3   **Round:** phase-1.5-cr   **ADDRESSED in 92c758f**: Added `SyncSettingsAuditBatchSink` type and `createJsonlBatchAuditSink()` factory; refactored `SettingsBridge.auditDelta` to collect ALL redacted entries first then dispatch through a single batch sink call (transactional — sink either commits all entries or none). Legacy per-entry sink path retained for hp-6obn callers. 5 new audit-bridge regression tests prove all-or-nothing rollback (`auditBatchSink: rejection rolls back ALL audit rows AND the in-memory delta`).
